# QA 教训(12 条)

> 来自 某 B 端中台项目 / 某营销后台项目 的真实 QA 踩坑

---

## 测试策略类

### #1 Layer 1 全绿 ≠ 没问题
**触发**:72 条 UI 存在性用例都过时。
**为什么**:UI 渲染通过 ≠ 数据流正确,batchId 类型错只有 Layer 3 能发现。
**规则**:4 层必齐,尤其 Layer 3 不能少。

### #2 Layer 3 是最精最贵但最该有
**触发**:做测试策略时。
**为什么**:核心 API payload 错 = 上线灾难,Layer 1 再多也挡不住。
**规则**:核心增删改查每个至少 1 条 Layer 3(拦 request 验 payload)。

### #3 回归用例要定期清理
**触发**:季度末或大版本前。
**为什么**:过期回归也是负担,跑得慢还误报。
**规则**:每季度清一次,超过 6 个月未命中的 bug 回归可删。

### #4 冒烟测试必须拒测,不心软
**触发**:开发提测但冒烟挂时。
**为什么**:心软 = 后面测试节奏崩,QA 成帮凶。
**规则**:冒烟 15 分钟跑不通 = 直接打回,不谈条件。

## 用例设计类

### #5 用例必须独立,不依赖执行顺序
**触发**:写 Playwright/Cypress 用例时。
**为什么**:依赖前一条的前置状态 = 并行跑或单独跑都会炸。
**规则**:每条用例自己造数据 + 清理,不假设其他用例结果。

### #6 用 API 造数据,不要点 UI
**触发**:需要特定场景的测试数据时。
**为什么**:UI 造数据慢且容易因 UI 改动失败。
**规则**:封装 test helpers 调 API 造数据,beforeEach 用。

### #7 PRD G-W-T 覆盖率核对
**触发**:用例产出完成时。
**为什么**:自己写用例容易漏 PRD 边角场景。
**规则**:提取 PRD 所有 Given-When-Then,逐条核对是否有用例。

### #8 selector 用 data-testid 最稳
**触发**:写 Playwright selector 时。
**为什么**:CSS class 常变,text 多语言会变,data-testid 最稳定。
**规则**:关键元素要前端加 data-testid,selector 优先选它。

## Bug 管理类

### #9 Bug 必须能复现才算真 Bug
**触发**:偶发问题。
**为什么**:开发无法复现 = 修不了 = 反复开关。
**规则**:试 3 次不复现 → 关闭并备注,新证据再开。

### #10 Bug 归属要当场判断
**触发**:不知道是前端还是后端时。
**为什么**:扯皮浪费时间,双方都不管 bug 烂掉。
**规则**:看 Network:字段名/parse 问题 → 前端;字段值/500 → 后端。

### #11 P0 不上线,没有"带病上"
**触发**:被业务 push 上线 deadline 时。
**为什么**:P0 带病 = 线上事故。
**规则**:P0 必修才能上,除非业务方书面接受降级上线(归档审计)。

## 自动化类

### #12 flaky 用例必须治理,不能 skip
**触发**:某用例 10 次成功 3 次。
**为什么**:skip 逃避 = CI 失去意义,该用例覆盖的真 bug 将来会漏。
**规则**:flaky 识别后 3 天内修,实在修不了标记 + 跟踪。

## Playwright 陷阱类

### #13 企业 SSO 环境用 storageState 复用 cookie
**触发**:为使用企业 SSO 的系统写 E2E 测试时。
**为什么**:企业 CAS/SSO 不支持代码注入 cookie,必须先 headed 模式手动登录后保存 storageState。
**规则**:用 `playwright codegen` 或脚本跑 headed 模式登录一次,保存 `storageState.json`,后续测试复用。

### #14 权限测试用哨兵值而非空数组
**触发**:测试无权限场景时。
**为什么**:`functionCodes = []` 空数组可能被前端当作"权限 API 未加载"处理,触发兜底全放行。
**规则**:用哨兵值 `['sentinel-has-perms-but-not-target']`,非空但不含目标权限码,准确触发无权限分支。

### #15 写 mock 前先 grep 确认前端实际读的字段名
**触发**:为 Playwright 测试写 mock 响应时。
**为什么**:前端 transform 层可能改名(如 `createUser` → `createdBy`),mock 错字段名 = 测试永远通过但验证的是错误路径。
**规则**:写 `route.fulfill` 前先执行 `grep -rn '字段名' src/`,确认前端实际消费的字段。

### #16 跨 spec 清 cookie 要单独放文件
**触发**:需要在某条用例中清除登录 cookie 测试登出场景时。
**为什么**:清 cookie 操作污染同文件其他 test 的 storageState,一起跑时其他用例失败。
**规则**:清 cookie 的 test 单独放一个文件,用 `test.use({ storageState: undefined })` 局部覆盖。

### #17 E2E 必须验证内容正确性,不只验存在性
**触发**:写列表、表单等页面的 E2E 用例时。
**为什么**:验证"有没有行"通过,不代表 batchId 或关键字段传对了,高危 bug 会漏。
**规则**:核心用例必须加内容断言,验证 API payload 中的关键字段值(用 `page.route` 拦截请求)。

## 测试分层类

### #18 三层测试模型分工要清晰
**触发**:设计测试策略时。
**为什么**:状态机逻辑塞 E2E 会 flaky;UI 显隐放 Service 单测漏渲染;混层导致维护成本翻倍。
**规则**:
- Service 单测:状态机、事务、字段计算(强断言)
- MockMvc 集成:HTTP 层、DTO 绑定、拦截器链(中断言)
- Playwright E2E:交互、视觉、权限显隐(只断不崩溃+核心可见)

### #19 功能改动后必须全链路检查
**触发**:修复某处展示或逻辑 bug 时。
**为什么**:只改一处,同类问题在其他页面/字段继续存在,测试走查不到。
**规则**:每次修复后 grep 同类问题,沿完整用户动线(列表→详情→编辑→提交)全链路验证。

### #20 诚实汇报测试结果,必须分四桶
**触发**:汇报测试执行结果时。
**为什么**:只报"X passed / Y failed"掩盖了 skip 和部分通过的情况,用户无法判断真实质量。
**规则**:结果必须分 ✅(通过) / 🖐(手工) / ⚠️(部分/依赖) / ❌(失败) 四类汇报。
