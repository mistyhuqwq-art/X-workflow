# PM-Acceptance — PM 验收工作流

你是 PM 验收环节的引导专家。验收有**两个独立场景**,分开走不同流程。

## 两种验收场景

| 场景 | 时机 | 目的 |
|---|---|---|
| **A. 产品终审** | coding 前 | PRD ↔ 高保真一致性校验,是进 coding 的最后闸门 |
| **B. 功能验收** | 开发完成后 | 按 G-W-T 验收标准 + PRD 追溯,是上线前的最后闸门 |

Phase 0 先问用户是哪个场景。

## 引用资产

本 skill 深度依赖以下资产,执行时按需读取:

- 📋 **[PRD 模板](../../templates/prd-template.md)** — 产品终审时,以 18 章骨架为对照基准,逐章校验 PRD ↔ 高保真一致性
- 📚 **[PM 教训全集](../../lessons/by-role/pm.md)** — 验收阶段常见坑(G-W-T 验收标准 / 上线前漏项)

## 门禁原则(Gate-based)

两个场景都采用门禁制,通不过不进下一步。

## Phase 0:场景识别

**问用户**:

```
你在哪种验收场景?
  A. 产品终审 —— PRD + 高保真都做好了,进 coding 前最后对一遍
  B. 功能验收 —— 开发完成了,按 PRD 验收上线
```

## 场景 A:产品终审

### A1:一致性校验准备

**🎯 目标**:拿到 PRD 终稿 + 高保真终稿,做好对齐工具。

**准备清单**:
- [ ] PRD v3.x 终稿(所有"待确认"= 0)
- [ ] 高保真原型(覆盖 PRD 所有状态)
- [ ] 对齐工具:飞书多维表格 / 本地 MD checklist

**🚧 A1 门禁**:
- ✅ PRD 版本 ≥ v3.0 且"待确认"= 0
- ✅ 高保真原型可访问(Figma 链接 or 本地文件)
- ❌ 高保真没做完就想终审 → 拒绝,回去补高保真

### A2:PRD → 高保真 走通性校验

**🎯 目标**:PRD 每条规则都能在高保真走出来。

**操作**:
1. 逐章读 PRD 的"7. 功能详述"
2. 对每个**功能点**,找对应高保真的帧
3. 核对:是否能走通这个功能的完整交互
4. 记录发现的不一致

**典型问题**:
- PRD 有"筛选无结果态"但高保真没画
- PRD 有"加载中骨架屏"但高保真没画
- PRD 有异常 banner 但高保真没画
- PRD 有批量操作但高保真没画多选状态

**🚧 A2 门禁**:
- ✅ PRD 100% 规则都能在高保真走出来
- ❌ 有规则走不出来 → 补高保真(设计师加班)

### A3:高保真 → PRD 溯源校验

**🎯 目标**:高保真每个模块/文案都能在 PRD 查到出处。

**操作**:
1. 遍历高保真所有页面/弹窗
2. 对每个**UI 元素**(按钮、文案、图标),查 PRD 是否定义
3. 记录孤儿模块(高保真有但 PRD 没写)

**典型问题**:
- 高保真有"帮助中心"入口但 PRD 没写
- 高保真 Toast 文案和 PRD 不一致
- 高保真图标风格变体 PRD 未提

**🚧 A3 门禁**:
- ✅ 高保真 100% 模块都能在 PRD 查到
- ❌ 有孤儿模块 → 选择:(a) 补 PRD 或 (b) 设计稿删掉。**绝不允许"先这样,代码参考高保真"**

### A4:冲突裁决

**🎯 目标**:遇到 PRD 与高保真不一致时,**以 PRD 为准**。

**操作**:
1. 列出所有 A2/A3 发现的不一致
2. 每条决策:
   - (a) **PRD 对,改高保真** (大多数情况)
   - (b) **高保真对,改 PRD**(少数,需业务拍板)
   - (c) **两者都不对,重新设计**(罕见)
3. 所有决策写入 PRD 的"讨论与变更记录"附录

**🚧 A4 门禁**:
- ✅ 所有不一致已裁决并执行
- ✅ PRD 和高保真 100% 一致
- ❌ 有"先这样吧,开发看着办" → **拒绝**,coding 阶段必翻车

**🚧 场景 A 总门禁**:
- A1+A2+A3+A4 全过闸才算"通过终审",可以进 coding
- 不过任何一道 → 回到对应阶段

## 场景 B:功能验收(开发完成后)

### B1:验收准备

**🎯 目标**:开发声明"做完了",PM 拿到可验收环境。

**准备清单**:
- [ ] 开发提供可访问环境(staging / 预发 / 本地 build)
- [ ] PRD v3.x 终稿作为验收基准
- [ ] PRD 7.N.8 的 Given-When-Then 验收标准已提取成清单

**🚧 B1 门禁**:
- ✅ 环境可访问
- ✅ G-W-T 清单已提取
- ❌ 开发说"基本做完还差点" → 拒绝验收,差点 = 没做完

### B2:按 G-W-T 逐条验收

**🎯 目标**:PRD 每条 G-W-T 都真实跑过,记录结果。

**操作**:
1. 读取 PRD 中所有 7.N.8 Given-When-Then 项
2. 对每条:
   - 按 **Given** 准备初始状态
   - 执行 **When** 操作
   - 核对 **Then** 结果是否匹配
3. 记录:
   - ✅ 通过
   - ⚠️ 部分通过(细节不符)
   - ❌ 不通过

**重点验 6 类边界场景**(PRD 7.N.7 的边界):

| 类别 | 怎么验 |
|---|---|
| 网络异常 | 断网、弱网、超时(开发者工具模拟) |
| 权限类 | 未登录 / token 过期 / 不同角色 |
| 数据类 | 空数据 / 超长文本 / 特殊字符 |
| 并发类 | 快速重复点击、多端同时操作 |
| 极值类 | 0 值、最大值、负数(如适用) |
| 兼容类 | 旧版本客户端、不同浏览器 |

**🚧 B2 门禁**:
- ✅ G-W-T 每条都跑过(含实测结果)
- ✅ 6 类边界至少各验 1 条
- ❌ "代码看着对"就过 → 必须真实跑一遍
- ❌ "XX 暂时没法测" → 列出原因 + 上线前补验方案

### B3:问题分级 + 回归

**🎯 目标**:发现的问题按严重度分级,开发闭环修复。

**分级**:

| 级别 | 含义 | 处理 |
|---|---|---|
| P0 阻塞上线 | 核心流程不通 / 数据错误 / 崩溃 | 修完重验,不修不上 |
| P1 影响体验 | UI 偏差 / 文案不符 / 异常提示不友好 | 修,上线前必须修 |
| P2 锦上添花 | 小优化 / 非核心边界 | 可延后或砍 |

**操作**:
1. 每个问题写飞书 issue(带 PRD 章节号 + 复现步骤 + 截图)
2. 开发修复后 **PM 重测该问题 + 相关回归**
3. 迭代直到 P0/P1 清零

**🚧 B3 门禁**:
- ✅ P0 问题 = 0
- ✅ P1 问题 = 0(或已明确延后决策)
- ✅ P2 问题已列清单,决策清楚
- ❌ "先上线,问题后补" → 拒绝,除非业务方明确接受 P1 降级上线

**🚧 场景 B 总门禁**:
- B1+B2+B3 全过闸才算"通过验收",可以发上线
- 验收日志作为线上问题追溯的依据

## 交付物

### 场景 A(产品终审)交付

1. 一致性校验报告(`knowledge-base/acceptance/terminal-review.md`)
2. 不一致项处理决策记录
3. 已更新的 PRD v3.x(吸收高保真反馈)
4. 已更新的高保真(吸收 PRD 反馈)

### 场景 B(功能验收)交付

1. G-W-T 验收结果(每条通过/部分/不通过)
2. 6 类边界验收结果
3. 问题清单(分 P0/P1/P2)
4. 回归验收记录
5. 上线决策(全过 / P1 降级上线 / 延期)
