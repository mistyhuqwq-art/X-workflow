# PM 教训(27 条)

> 来自 某 B 端中台项目 / 某营销后台项目 / 某 B 端中台项目 / MVP 社区项目的真实 PM 踩坑

## 物料准备类

### #1 物料不足比物料不准更致命
**触发**:启动新需求时。
**为什么**:AI 按你给的材料推理,缺材料 = 推理建立在假设上。物料补齐前不要动笔。
**规则**:3 类物料(参照/数据/继承清单)全齐才动 PRD。

### #2 数据口径必须早对齐,越早越好
**触发**:拉线上数据做决策时。
**为什么**:一处 UV 一处 PV 混用,下游 PRD 每条决策都锚错,review 阶段要求重新拉数。
**规则**:出手前先对齐口径,显式注明"这里 UV"/"那里 PV"。

### #3 竞品和"想做的"必须分层标注
**触发**:收集竞品截图时。
**为什么**:截图混入主观想做的功能,AI 把"想做的"当"竞品已有"输出,事实核查翻一倍工。
**规则**:竞品截图 vs 需求灵感分 2 个文件夹,不混放。

### #4 继承清单必须单独成章
**触发**:改版项目(不是全新项目)时。
**为什么**:只写新功能不写继承,研发默认全部新建,估期翻倍。
**规则**:PRD 独立一章"现有逻辑继承清单",继承/修改/不做三列列清。

## 三步对话类

### #5 不能跳过苏格拉底直接拆解
**触发**:有初步需求想直接进第一性时。
**为什么**:问题没问对时,拆得再细都是错方向。
**规则**:强制先问 5 个真问题(真实目标/不做什么/痛点/成功/失败标准),再进第一性。

### #6 真实目标不能是"提升 X%"
**触发**:写需求目标时。
**为什么**:提升 X% 是结果指标,不是真实目标。真实目标是"用户的 Y 问题"。
**规则**:目标一句话是"解决用户/业务的 XX 问题",不是数字。

### #7 必要条件必须产品拍板
**触发**:用 AI 拆需求时。
**为什么**:让 AI 自己定义"必要条件",AI 会编合理但非必要的条目(如"必须个性化推荐")。
**规则**:AI 只给候选,必要条件由 PM/业务拍板。

### #8 砍一刀回头看状态机
**触发**:用奥卡姆砍模块时。
**为什么**:单个模块看能砍,但砍了破坏状态机闭环。
**规则**:每砍一刀,回头检查状态机是否仍闭环。

## PRD 产出类

### #9 18 章骨架不允许缺章
**触发**:v1.0 写初稿时。
**为什么**:缺章 = 后期某类问题没想,review 时爆出来返工更大。
**规则**:每章必填,没内容写"待确认/待补充/待法务"占位。

### #10 v3.0 前所有"待确认"必须清零
**触发**:PRD v3.x 准备提 review 时。
**为什么**:带"待确认"进 review,技术 3 端会各自假设不同解,review 结论混乱。
**规则**:"待确认"清单 + 责任人 + deadline,v3.0 前全部收敛。

### #11 凡标"必须澄清"一律业务拍板
**触发**:PM 不知道答案的关键项。
**为什么**:PM 自行推断核心规则 → 建在假设上 → 返工代价极大。
**规则**:AI 和 PM 都不能替业务答"必须澄清"项,推业务方拍板。

**子规则 · 推不动时的升级路径**(2026-04-24 合入):
- **触发**:业务方对"必须澄清"项拖延不回。
- **为什么**:某营销后台 终验 3 项阻塞全是"必须澄清"没收敛,PM 独自催等于推不动。
- **规则**:每条"必须澄清"写明 `deadline + 升级路径`(示例:"24h 不回 → 上升级 IC / 研发负责人")。这避免 PM 被动等待,让沉默也有成本。

## 评审协作类

### #12 三端 review 必须联合,不能串行
**触发**:PRD 送审时。
**为什么**:服务端通过 ≠ 客户端通过,各自理解不同侧重(数据接口 vs UI vs 数据路径)。
**规则**:服务端 + 客户端 + 测试必须联合 review,发现冲突当场裁决。

### #13 设计 review 和技术 review 必须并行
**触发**:PRD 和高保真同步产出时。
**为什么**:串行 = 设计改完高保真,技术说做不了,设计再改。
**规则**:两边并行,核心交互提前同步技术可行性。

### #14 大文档让 AI 自行拆分低效
**触发**:PM 想让 AI 读长文档时。
**为什么**:大文档拆分费时费力,生成质量也不稳。
**规则**:改为截图每个模块让 AI 读取,速度从半天降到 1 小时。

## 验收 / 终审类

### #15 PRD 和高保真不一致时,以 PRD 为准
**触发**:终审发现对不齐时。
**为什么**:多信息源 = 研发 / QA / 上线后各实现一版,上线必不一致。
**规则**:PRD 是单一信息源,不一致时改高保真,不是改 PRD。

## 迭代节奏类

### #16 PRD 变更必须同步设计稿
**触发**:联调或 review 后 PRD 有任何改动时。
**为什么**:PRD 改了但设计稿没跟上 → 研发/QA 各自按不同版本实现,上线必出差异。
**规则**:PRD 每次迭代后,同日检查设计稿是否对齐,不一致立即标 [STALE] 并更新。

### #17 多角色并行时主动派任务,不等用户提醒
**触发**:同时推进需求、设计、开发多条线时。
**为什么**:等用户逐步推动 = 浪费协调等待时间,1 人 PM 全流程容易遗漏。
**规则**:每完成一个里程碑,主动识别下一步可并行的任务并拆解执行,不等用户安排。

### #18 提出假设时先自己验证,不丢给用户
**触发**:分析问题时想到可能的根因。
**为什么**:丢给用户验证 = 浪费用户时间,AI 辅助的核心价值是主动缩小问题空间。
**规则**:凡是能跑命令/读日志/读代码验证的假设,先验再报告,用"已验证:是/否"格式汇报。

## 设计决策类

### #19 混合交付路线比全量高保真更务实
**触发**:项目周期紧张,设计工作量大时。
**为什么**:全量高保真耗时且边际收益递减,简单页面直接开发反而更快交付。
**规则**:复杂核心页面出高保真,简单配置页面直接出线框 + 组件规范给开发。

### #20 组件库先 review 再出原型,不能同步进行
**触发**:开始设计原型阶段时。
**为什么**:组件库未稳定就出原型 → 原型引用错误组件 → 交付后开发返工。
**规则**:Phase 1(组件库)review 通过后才进 Phase 2(原型),两阶段串行,不并行。

## 需求评审类

### #21 AI 生成 PRD 不会主动发现后端已有约束
**触发**:基于已有后端代码开发新功能时。
**为什么**:AI 按 PRD 描述推理,不会主动读后端代码发现枚举/字段不一致,这类 gap 在联调才爆出。
**规则**:PRD 完成后,与后端对照关键枚举和字段名一轮,标注差异。

### #22 检查报告禁止模糊表达
**触发**:AI 或人工出检查/review 报告时。
**为什么**:"基本一致""未发现重大问题"是无效信息,无法判断是否真的检查了。
**规则**:每项检查必须给出确定结论:✅ 一致 / ❌ 不一致(具体差异) / 🖐 未检查(原因)。

### #23 Review 类任务必须先汇报再修复
**触发**:执行 review 或检查任务时。
**为什么**:边查边改会丢失"发现了什么"的轨迹,后续无法核对是否修复到位。
**规则**:先输出完整发现报告,等确认后再执行修复,不能合并成一步。

## AI 协作类

### #24 字段映射必须在并行开发前三方对齐
**触发**:后端契约未冻结想让前端先 Mock 并行开发时。
**为什么**:纯 Mock 开发联调暴露 40% 字段差异,6 天修 41 bug（某营销后台 实录）。
**规则**:PRD v3.0 前必须补"字段映射章节",前端↔后端↔PRD 三方对齐签字,否则不允许并行开发。

### #25 AI 候选完成 ≠ 人工正式完成
**触发**:AI 协作项目准备宣布"完成"时。
**为什么**:AI 说完成,人测必发问题（某 B 端中台项目 Day 12→13 / 某营销后台 Day 2→3 皆如此）。
**规则**:AI 自测 = "候选完成",人工走查 = "正式完成"；PM 排期必预留 20% 工时给人工验收。

## 决策质量类

### #26 关键决策必须带"显形时间预估"
**触发**:做关键决策时（范围/技术路线/Mock 策略等）。
**为什么**:错误决策 2 天~3 周才显形,没锚点就无法按时复盘。
**规则**:决策记录字段增加"预期显形时间"（即时 / X 天 / X 周），到点必回头 check 是否如预期。

### #27 混合路线决策必须放 Phase 0
**触发**:工时预估 > 1.5× budget,或明显有简单页面可直出时。
**为什么**:某营销后台 Phase 10e 才决策混合路线,已烧 3 天；某 B 端中台项目 Day 6-7 才分流,同类错误。
**规则**:Phase 0（需求对齐阶段）就评估"是否混合路线"（全量高保真/线框+组件规范/直接前端），不等撞南墙才调整。
