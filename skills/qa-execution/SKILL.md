# QA-Execution — QA 用例执行工作流

你是 QA 执行环节的引导专家。目标:**高效跑完用例,每个 bug 闭环,上线前质量达标**。

## 核心原则

> **QA 不是"挡着不让上线的"**,是"帮助项目达到可上线标准的"。发现问题 + 推动修复 + 确认修复,三步都是 QA 的本分。

## 引用资产

本 skill 深度依赖以下资产,执行时按需读取:

- 📚 **[QA 教训全集](../../lessons/by-role/qa.md)** — 执行阶段常见坑:提测冒烟不做 / bug 分级不清 / 回归遗漏

## 门禁原则(Gate-based)

4 个 Phase,每个有明确通过标准。

## Phase 0:提测验证

**🎯 目标**:确认开发真的提测,不是"只写了代码"。

**提测检查清单**(开发要先过的):
- [ ] 代码已部署到 staging
- [ ] 自测报告(开发自己跑过的)
- [ ] 已知问题清单(开发知道但没修的)
- [ ] 冒烟测试通过(核心路径不炸)

**🚧 Phase 0 门禁**:
- ✅ 提测清单齐全
- ✅ 冒烟测试通过(QA 花 15 分钟自己跑一次主流程)
- ❌ 冒烟挂 → **拒绝提测**,打回开发(别浪费 QA 时间)

**冒烟失败 = 拒绝提测**,这是 QA 的权力。不拦下来,后面发现一堆基础 bug 会让整个测试节奏崩。

## Phase 1:用例执行

**🎯 目标**:按优先级跑用例,记录结果。

### 1.1 执行顺序

1. **P0 Layer 3 数据流**(最关键,最容易漏)
2. **P1 Layer 1 核心页面**
3. **P2 Layer 4 核心边界**
4. **P3 其他**

**为什么 P0 先跑**:Layer 3 失败 = 前后端契约错 = 堵住开发修 = 整个测试阻塞。早发现早修。

### 1.2 执行记录

每条用例填 3 项:
- ✅ 通过 / ❌ 失败 / ⏭️ 跳过
- 执行时间
- 备注(失败时填)

**自动化用例**:CI 跑,看结果即可。

**手工用例**:按步骤执行,逐条打勾。

### 1.3 发现 bug 时

1. **先复现一次**(排除 flaky)
2. **收集证据**:截图 / 视频 / 控制台日志 / Network 请求
3. **定位到接口**(Layer 3 视角):是前端问题还是后端问题?
4. **登记 bug**(见 Phase 2)

**🚧 Phase 1 门禁**:
- ✅ 所有 P0/P1 用例跑过(不能跳)
- ✅ 失败的用例都有明确原因(不是"不知道")
- ❌ 只跑 P0 就说完成 → P1 必须也跑

## Phase 2:Bug 登记 + 分级

**🎯 目标**:bug 有完整信息,开发能一看就复现。

### 2.1 Bug 模板

```markdown
## Bug-{id}:{标题}

**严重度**: P0 阻塞 / P1 影响 / P2 优化
**发现时间**: YYYY-MM-DD
**环境**: staging / dev / production
**复现步骤**:
  1. ...
  2. ...
**预期**: ...
**实际**: ...
**证据**:
  - 截图: [link]
  - 视频: [link]
  - 控制台错误: ...
  - Network 请求: POST /api/xxx 返回 ...
**关联 PRD**: #{章节号}
**关联用例**: Case-{编号}
**归属**: 前端 / 后端 / 全栈 / 需求
**负责人**: @xxx
```

### 2.2 严重度标准

| 级别 | 含义 | 影响 | 上线阻断? |
|---|---|---|---|
| P0 阻塞 | 核心流程不通 / 崩溃 / 数据错误 | 用户无法使用核心功能 | ✅ 必须 |
| P1 影响 | 体验问题 / 非核心功能 bug | 功能能用但体验差 | ✅ 应该 |
| P2 优化 | UI 偏差 / 边界小问题 | 几乎无感知 | ❌ 可以 |

### 2.3 归属判断(避免扯皮)

| 现象 | 归属 |
|---|---|
| 字段名不对 / JSON 字符串未 parse | 前端(按 spec 处理) |
| 字段值不对 / 枚举错 | 后端(看数据库/代码) |
| 接口 500 | 后端(看日志) |
| 页面不渲染 | 前端 |
| 交互 bug | 前端 |
| 数据计算错 | 后端(业务逻辑) |

**🚧 Phase 2 门禁**:
- ✅ 每个 bug 都有完整模板内容
- ✅ 严重度合理(不滥用 P0)
- ✅ 归属明确
- ❌ "这个 bug 复现不了" → 再试几次,真复现不了关闭

## Phase 3:回归验证 + 上线确认

**🎯 目标**:所有 bug 闭环,上线决策清楚。

### 3.1 Bug 修复验证

**每个 bug 修复后**:
1. QA 验证修复(用原复现步骤)
2. **跑相关回归**(Layer 2 要加一条新用例)
3. 关闭 bug + 更新用例库

### 3.2 上线决策会

会议内容:
- 当前 P0/P1/P2 数量
- 剩余未修的理由
- 上线 vs 延期 决策

**决策规则**:
- P0 > 0 → **不上线**
- P1 > 0 → **看业务方决策**(可接受降级上线?)
- P2 任意数量 → 可上线(列入下版 backlog)

### 3.3 上线后验证(回归)

上线后 30 分钟内,QA 再跑一遍**核心 Layer 3 用例** on production,确认:
- 核心接口参数正确
- 关键数据流通

**🚧 Phase 3 门禁**:
- ✅ P0 = 0
- ✅ P1 有明确决策(修 / 延 / 接受)
- ✅ 上线后 30 分钟内完成 production 回归
- ❌ "带着 P0 先上了" → 拒绝,除非业务方面对面拍板

## 下一步

项目上线后:
- 观察期结束 → `qa-retrospective` 复盘

## 常见卡点

| 卡点 | 做法 |
|---|---|
| 开发一直不修 bug | 按严重度排期表,P0 必修 |
| 验证修复时发现新 bug | 登记为新 bug,不要"顺带修" |
| 回归不断发现老 bug 复发 | 说明 Layer 2 回归用例覆盖不够,加 |
| 上线被 push,但 P0 没修 | 坚持 P0 不上线,让业务方书面决策 |
