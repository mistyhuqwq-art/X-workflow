## 接口设计类(前端联调最痛)

### #1 陷阱字段必须在 spec 里显式标注
**触发**:写接口 spec 时。
**为什么**:前端无法从字段名推断是 string 还是 object / JSON 字符串。
**规则**:
- ID 字段明确 string 还是 number
- JSON 字符串标"string(JSON)" + 内部结构
- 枚举字段列出全部枚举值
- 时间字段标时区(UTC / 本地)和格式

### #2 命名风格一旦选定全项目统一
**触发**:项目启动接口设计。
**为什么**:snake_case 和 camelCase 混用 = 前端必踩雷。
**规则**:选一种 + 写在项目规范,不允许混用。

### #3 错误码必须多级,不能只 0 和 -1
**触发**:设计错误返回时。
**为什么**:前端只能根据 code 决定跳登录/提示/重试,模糊错误码无法分流。
**规则**:至少 4 类:400xxx 参数 / 401/403 权限 / 500xxx 业务冲突 / 500xxx 系统。

### #4 枚举值变更必须走变更流程
**触发**:项目中途想改枚举含义。
**为什么**:前端已写死某枚举值,后端改 = 前端悄悄翻车。
**规则**:枚举改动 → 通知前端 + 加新值不删老值 + 考虑兼容期。

## 数据库设计类

### #5 JSON 字段用 VARCHAR + TypeHandler
**触发**:需要存半结构化数据。
**为什么**:MySQL JSON 类型查询慢,TypeHandler 自动序列化方便。
**规则**:VARCHAR 存 + TypeHandler 自动 parse,不要直接存 JSON 类型。

### #6 索引必须和业务查询对齐
**触发**:建表时。
**为什么**:业务跑起来了发现慢查询,此时加索引要锁表重建。
**规则**:建表时就看业务 WHERE/ORDER BY,提前建索引。

### #7 migration 必须有回滚方案
**触发**:写 SQL migration 时。
**为什么**:上线出问题无法回滚 = 只能硬撑 + 紧急 hotfix。
**规则**:每个 UP 都要对应 DOWN,上线前 test 过回滚流程。

### #8 冗余 vs 关联看查询频率
**触发**:表关系设计时。
**为什么**:高频查询做冗余 = 读快写慢,低频做关联 = 读慢写快。
**规则**:高频读(如首页)冗余,低频(如报表)JOIN。

## 编码类

### #9 Entity 不直接返前端
**触发**:写 Controller 时。
**为什么**:Entity 字段暴露全,命名也不符契约。
**规则**:Entity → DTO 转换,只给前端需要的字段。

### #10 不用 Object/Map 做响应
**触发**:返回结构不定的数据时。
**为什么**:无类型,前端无法类型推导 + 后端后续改字段无提醒。
**规则**:每个接口独立 ResponseDTO,即使只有 1 个字段。

### #11 @ControllerAdvice 统一异常处理
**触发**:业务逻辑 throw 异常时。
**为什么**:未处理异常直接到 Tomcat → 前端收到 500 空 body。
**规则**:全局 @ControllerAdvice 转成统一 { code, message } 返回。

### #12 SQL 不用字符串拼接
**触发**:写复杂查询时。
**为什么**:SQL 注入,测试环境可能没暴露生产会炸。
**规则**:用 MyBatis 参数绑定,或 JPA Specification。

## 联调配合类

### #13 staging 环境稳定是后端责任
**触发**:前端开始联调时。
**为什么**:环境挂 = 前端等一天,整个团队节奏崩。
**规则**:保证 staging 可访问 + 测试数据充足,前端有问题 30 分钟内响应。

### #14 curl/Postman 自测 > 单元测试
**触发**:声明"接口做完了"时。
**为什么**:单元测试用假数据,真实环境可能有 DB/CORS/鉴权问题。
**规则**:所有接口都要真实 curl 过,不靠单测过就算完成。

### #15 日志要带 traceId
**触发**:线上接口报错时。
**为什么**:没 traceId = 日志海里捞针。
**规则**:MDC 传 traceId,每条日志带,前端报 bug 附 traceId 快速定位。
