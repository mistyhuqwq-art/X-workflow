Why I deleted 40%
of my startup's code.

三个月前我们从 12 万行砍到 7.2 万行,什么都没少做,反而快了。这不是 "less is more" 的口号,是一次具体的重构笔记。

写给还在给产品 "加功能" 的年轻工程师——你可能在加的,是未来要维护的债。

findings

三个被 删除 后才发现的事实

这些不是节省工时的 trick,是关于"复杂度真实来源"的观察。

  • 01 防御式代码占了 30% —— 那些 try/catch、null check、fallback,大多数永远不会触发。当我们去掉并认真思考边界时,真正需要防御的场景其实不到 6 处。
  • 02 "配置化" 是过度设计的温床 —— 我们有 14 个 feature flag,其中 11 个从来没被关过。把它们硬编码后,代码变得可读,bug 变得可追。
  • 03 抽象比重复危险得多 —— 3 个相似但不同的函数被强行抽成 1 个通用的,随后一年里每次修改都要改 3 个 caller。恢复成 3 个独立函数后,每个都更清晰、改动更局部。
The best code is the code you didn't have to write. The second best is the code you had the courage to delete. — old engineering wisdom, restated
before · after

一个具体的例子

把一段 28 行的 "灵活" 实现,换成 4 行的直白实现。

// BEFORE · 28 lines · "extensible" abstraction function processItem(item, options = {}) { const validators = options.validators || defaultValidators; const transformers = options.transformers || []; const errorHandler = options.onError || logAndContinue; // ... 22 more lines of indirection } // AFTER · 4 lines · does exactly what we need function processItem(item) { if (!item.id) throw new Error('missing id'); return { ...item, processed_at: Date.now() }; }

一年下来 `processItem` 只被调用了 3 处,全都是用默认配置。所谓的 "extensibility" 完全没人用过——却让代码读起来像看谜。删掉之后,新同事理解这段逻辑的时间从半小时变成 30 秒。

conclusion

所以,什么时候该删?

不是"有空就删"——是三个信号出现时,立刻删:

  • 01 没人记得它为什么存在。 git log / blame 显示超过 6 个月没人改,且没人能说出它解决的问题——高概率是僵尸代码。
  • 02 修 bug 总要"顺便改它"。 说明它耦合过深,且它提供的抽象已经不匹配现状。删掉它,bug 的修复路径反而变短。
  • 03 它覆盖不到的 edge case 越来越多。 (这条本来在这,但我决定删掉——写成这样已经是过度抽象了。)
  • 04 它的名字说不清。 被命名为 `Util` / `Helper` / `Common` / `Manager` 的模块,80% 的情况下应该被解散成更具体的函数。
