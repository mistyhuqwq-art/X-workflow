# QA 教训(12 条)

> 来自 某 B 端中台项目 / 某营销后台项目 的真实 QA 踩坑

---

## 测试策略类

### #1 Layer 1 全绿 ≠ 没问题
**触发**:72 条 UI 存在性用例都过时。
**为什么**:UI 渲染通过 ≠ 数据流正确,batchId 类型错只有 Layer 3 能发现。
**规则**:4 层必齐,尤其 Layer 3 不能少。

### #2 Layer 3 是最精最贵但最该有
**触发**:做测试策略时。
**为什么**:核心 API payload 错 = 上线灾难,Layer 1 再多也挡不住。
**规则**:核心增删改查每个至少 1 条 Layer 3(拦 request 验 payload)。

### #3 回归用例要定期清理
**触发**:季度末或大版本前。
**为什么**:过期回归也是负担,跑得慢还误报。
**规则**:每季度清一次,超过 6 个月未命中的 bug 回归可删。

### #4 冒烟测试必须拒测,不心软
**触发**:开发提测但冒烟挂时。
**为什么**:心软 = 后面测试节奏崩,QA 成帮凶。
**规则**:冒烟 15 分钟跑不通 = 直接打回,不谈条件。

## 用例设计类

### #5 用例必须独立,不依赖执行顺序
**触发**:写 Playwright/Cypress 用例时。
**为什么**:依赖前一条的前置状态 = 并行跑或单独跑都会炸。
**规则**:每条用例自己造数据 + 清理,不假设其他用例结果。

### #6 用 API 造数据,不要点 UI
**触发**:需要特定场景的测试数据时。
**为什么**:UI 造数据慢且容易因 UI 改动失败。
**规则**:封装 test helpers 调 API 造数据,beforeEach 用。

### #7 PRD G-W-T 覆盖率核对
**触发**:用例产出完成时。
**为什么**:自己写用例容易漏 PRD 边角场景。
**规则**:提取 PRD 所有 Given-When-Then,逐条核对是否有用例。

### #8 selector 用 data-testid 最稳
**触发**:写 Playwright selector 时。
**为什么**:CSS class 常变,text 多语言会变,data-testid 最稳定。
**规则**:关键元素要前端加 data-testid,selector 优先选它。

## Bug 管理类

### #9 Bug 必须能复现才算真 Bug
**触发**:偶发问题。
**为什么**:开发无法复现 = 修不了 = 反复开关。
**规则**:试 3 次不复现 → 关闭并备注,新证据再开。

### #10 Bug 归属要当场判断
**触发**:不知道是前端还是后端时。
**为什么**:扯皮浪费时间,双方都不管 bug 烂掉。
**规则**:看 Network:字段名/parse 问题 → 前端;字段值/500 → 后端。

### #11 P0 不上线,没有"带病上"
**触发**:被业务 push 上线 deadline 时。
**为什么**:P0 带病 = 线上事故。
**规则**:P0 必修才能上,除非业务方书面接受降级上线(归档审计)。

## 自动化类

### #12 flaky 用例必须治理,不能 skip
**触发**:某用例 10 次成功 3 次。
**为什么**:skip 逃避 = CI 失去意义,该用例覆盖的真 bug 将来会漏。
**规则**:flaky 识别后 3 天内修,实在修不了标记 + 跟踪。
