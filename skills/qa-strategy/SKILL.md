# QA-Strategy — QA 测试策略设计工作流

你是 QA 测试策略环节的引导专家。目标:**让 QA 不只是"跑用例的",而是"和前端/后端协同的质量架构师"**。

## 核心原则

> **测试不是"找 bug",是"系统性保障质量"**。找 bug 是手段,保障质量是目的。

**某 B 端中台项目 实战教训**:
- 72 条 Layer 1(UI 存在性)全绿,但 batchId 传错(string→number)只有 Layer 3 能发现
- **Layer 1 全绿 ≠ 没问题** — 分层策略缺一不可

## 4 层测试分层架构(核心资产,跨项目复用)

| 层级 | 目的 | 发现的 bug 级别 |
|---|---|---|
| **Layer 1:UI 存在性** | 页面/组件/路由能渲染 | 组件缺失、路由错误 |
| **Layer 2:回归测试** | 已修 bug 不复发 | 回归 bug |
| **Layer 3:端到端数据流** ⭐ | API 参数正确性 | **字段映射错误**(联调翻车根源) |
| **Layer 4:异常边界** | 错误处理、边界条件 | 健壮性问题 |

**关键**:**Layer 3 是最精最贵但最该有的**。金字塔倒过来看,Layer 3 虽然用例少但效益最高。

## 门禁原则(Gate-based)

4 个 Phase,每个有明确产出和通过标准。

## Phase 0:前置对齐

- [ ] PRD v3.x 终稿(含 G-W-T 验收标准)
- [ ] 接口 spec(后端冻结版)
- [ ] 开发排期(知道什么时候能提测)
- [ ] 测试环境准备状态(staging 可用?)
- [ ] 既有测试资产(回归用例库)

**🚧 Phase 0 门禁**:
- ✅ PRD / 接口 spec 都是终稿
- ❌ PRD 还在改 → 等定稿再来

## Phase 1:测试范围和优先级

**🎯 目标**:明确测什么 + 不测什么 + 各自的优先级。

### 1.1 测试范围

| 范围 | 是否本期 | 负责层级 |
|---|---|---|
| 新功能功能测试 | ✅ | Layer 1 + Layer 3 |
| 新功能边界测试 | ✅ | Layer 4 |
| 历史功能回归 | ✅ 核心路径 | Layer 2 |
| 性能测试 | 看项目 | 专项 |
| 安全测试 | 看项目 | 专项 |
| 兼容性测试 | 看项目 | Layer 4 |

### 1.2 优先级

| 优先级 | 测什么 | 放弃成本 |
|---|---|---|
| P0 Layer 3 数据流 | 核心业务接口 payload | 放弃 = 上线字段 bug |
| P1 Layer 1 核心页面 | 主要页面渲染 | 放弃 = 基础回归失效 |
| P2 Layer 4 核心边界 | 6 类边界主要场景 | 放弃 = 客诉集中 |
| P3 Layer 2 + Layer 4 非核心 | 其他 | 可接受 |

**🚧 Phase 1 门禁**:
- ✅ 每层都有范围说明(不是只写 Layer 1)
- ✅ 优先级清晰可排期
- ❌ "所有都要测" → 拒绝,必须分层优先级

## Phase 2:工具选型和环境

**🎯 目标**:选定测试工具栈,环境就绪。

### 2.1 工具栈

| 目的 | 推荐工具 | 备注 |
|---|---|---|
| E2E(前端) | Playwright | 跨浏览器、拦截请求 |
| 接口测试(后端) | Postman / Newman / HTTP 文件 | |
| 性能压测 | JMeter / Gatling / k6 | |
| Mock | YAPI / Mock Service Worker | 前后端联调时 |
| CI 集成 | GitHub Actions / Jenkins | 用例自动跑 |
| 测试管理 | 飞书多维表格 / Xmind 脑图 | 用例追溯 |

### 2.2 测试环境

| 环境 | 用途 | 数据 |
|---|---|---|
| 本地 | 开发自测 | Mock |
| dev | 内部联调 | 测试数据 |
| staging | QA 测试 + 预发 | 脱敏生产数据 |
| production | 上线 | 真实数据 |

**🚧 Phase 2 门禁**:
- ✅ 工具栈选定
- ✅ staging 环境可访问
- ❌ "工具先用着,有问题再换" → 拒绝,改工具成本大

## Phase 3:用例量和人力预估

**🎯 目标**:预估用例数量 + 测试时长 + 人力分配。

### 3.1 用例量预估(基于 某 B 端中台项目 + 某营销后台 经验)

**经验公式**:

| 层级 | 经验用例/功能模块 | 示例(15 个功能) |
|---|---|---|
| Layer 1 | ~5 条 | 75 条 |
| Layer 2 | ~0.5 条(由 bug 驱动) | ~10 条 |
| Layer 3 | ~1 条 | 15 条 |
| Layer 4 | ~2 条 | 30 条 |
| **合计** | | **~130 条** |

**补充**:
- 大型 B 端中台项目可以达到 **500-1000 条**(某营销后台 887 条)
- 关键是 **Layer 3 不能少**,Layer 1 再多也掩盖不了 Layer 3 空白

### 3.2 时长预估

| 阶段 | 时长 |
|---|---|
| 用例设计 | 3-5 天(看复杂度) |
| 用例编写 + 跑通 | 3-7 天 |
| 提测后功能测试 | 3-5 天 |
| 回归 + 修复验证 | 2-3 天 |

**🚧 Phase 3 门禁**:
- ✅ 用例量 + 时长预估有依据(不是拍脑袋)
- ✅ 人力分配明确
- ❌ "测不完" → 砍低优先级,不是全测

## Phase 4:策略 review + 交付

**🎯 目标**:策略和 PM / 前端 / 后端 对齐,达成共识。

**Review 要点**:
- [ ] PM 确认范围和优先级覆盖需求
- [ ] 前端/后端确认测试环境和接口对接方式
- [ ] 架构组确认工具选型
- [ ] 时间安排与上线日期对齐

**交付物**:
- `knowledge-base/qa/test-strategy.md` — 测试策略文档
- 或飞书文档

**🚧 Phase 4 门禁**:
- ✅ 三方 review 通过
- ✅ 策略文档共享位置固定
- ❌ 一方挂起 → 不能进 qa-cases 阶段

## 下一步

策略通过后:
- 调用 `qa-cases` skill 产出具体用例
- 并行准备环境和数据

## 常见卡点

| 卡点 | 做法 |
|---|---|
| PRD 太粗测不了 | 回 PM 补细节(不是 QA 猜) |
| 用例太多测不完 | 按优先级砍 P3,P0/P1 必保 |
| 环境不稳定 | 先解决环境,不稳定跑用例 = 白跑 |
| 回归用例太老没用 | 每季度清理一次(过期回归也是负担) |
