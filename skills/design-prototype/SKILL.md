# Design-Prototype — 设计原型组装工作流

你是原型组装环节的引导专家。目标:**先用 Mockup 对齐结构,再用高保真还原细节,每个页面覆盖 10 类必备帧、0 手搭视觉元素**。

## 核心原则

> **设计稿不是给设计师自己看的,是给研发+QA 看的**。研发看了就能实现,QA 看了就能测,才算合格。
>
> **结构错误必须在低保真阶段暴露**。高保真阶段改结构 = 改组件 + 改布局 + 改连线,成本是 Mockup 阶段的 10 倍以上。

## 门禁原则(Gate-based)

4 个 Phase,每个有明确产出和通过标准。Phase 1(Mockup)是 Phase 2(高保真)的强制前置。

## Phase 0:前置检查

- [ ] `design-alignment` 场景 A 已完成(对齐 5 项)
- [ ] `design-system` 已完成(组件库可用)
- [ ] PRD v3.x 终稿可访问
- [ ] 页面清单已列(做哪几个页面)

**🚧 Phase 0 门禁**:
- ✅ 组件库已 review 通过
- ❌ 组件库没 review 就直接出原型 → 拒绝,原型必返工(实测教训)

## Phase 1:Mockup 多版迭代 — 结构先对齐(强制门禁)

**🎯 目标**:**在投入高保真前,用 Mockup 和 PM + 前端迭代对齐信息架构 + 行为规则**。一般需要 3-5 轮,每轮 15-30 分钟,结构/行为错误在这里全部暴露。

### 核心认知:Mockup 不是一版交付,是多版迭代

实战经验:**一版 ASCII 就拿去敲定 = 天真**。Mockup 从 v1 走到 vN(通常 v3-v5)才成熟。每一版聚焦不同维度:

| 版本 | 关注点 | 典型产出 |
|---|---|---|
| **v1** | 区块划分 + 字段清单 + 必填 | 最简框图,只画有什么 |
| **v2** | 字段联动 + 数量关系(A 字段驱动 B) | 在 v1 基础上标联动箭头/注释 |
| **v3** | 行为说明(异步同步/默认值/系统自动分配) | 加 `ℹ️` 提示行,写清规则 |
| **v4+** | 示例数据 + 未选态 + 成熟态 | 填真实候选值,区分「已选/未选」 |

**每一版都是一轮对齐**,PM/前端看到新版本就能提出下一层问题,层层收敛。

### 精细度边界:信息可以密,视觉不能精

**允许**(信息精细):真实示例数据、行为说明、系统规则提示、未选态区分、字段联动注释。
**禁止**(视觉精细):配色、阴影、圆角、图标美化、字体样式、具体间距 —— 这些一上就退化成高保真,失去试错价值。

判断标准:**能不能用纯文本(ASCII)或灰底框画出来?能 → 信息维度 → OK。必须 Figma 上色调样式才能表达? → 视觉维度 → 出圈了**。

### Mockup 形式(选一,够用就行)

| 形式 | 适用场景 | 工具 |
|---|---|---|
| ASCII 框图 | 表单、配置面板、简单列表 | 纯文本,飞书/issue 直接贴 |
| 纸面草图 | 页面级结构、多区域布局 | 手绘拍照 |
| Figma 低保真框 | 复杂多分组/多 tab | Figma 灰底空矩形 + 文字标签 |

**v4 多了什么**:
- `ℹ️` 行为说明(适用位置的同步规则、参数值的分配规则)
- 示例数据(春节首焦 01/20、详情页A版 01/22)—— 让 PM 看到真实语境
- 未选态(`└(未选)`)—— 明确空值时怎么显示

### 对齐流程(3-5 轮,每轮 15-30 分钟)

1. 设计师出 v1 → 发 PM + 前端(不开会,异步看)
2. 反馈形式:结构 OK/不 OK + 具体问题(字段该加/该删、联动不对、分组逻辑)
3. 设计师吸收反馈出 v2,聚焦联动关系 → 再发一轮
4. 继续迭代,每轮聚焦一个维度(结构 → 联动 → 行为 → 示例数据)
5. 直到**三方都说「没问题了」**才进入 Phase 2 做高保真

**常见轮数参考**:简单表单 2-3 轮,复杂配置页 4-6 轮。**第一版就拿去当终稿 = 危险信号**。

**🚧 Phase 1 门禁**:
- ✅ Mockup 至少迭代到 vN,PM + 前端都回复「结构 + 行为都 OK」(飞书留痕)
- ✅ Mockup 覆盖所有需要做高保真的页面(每个页面独立一套)
- ✅ 行为说明、字段联动、默认值规则都已标出
- ❌ "我自己看 v1 没问题直接出高保真" → 拒绝,结构风险由设计师一人扛,翻车率极高
- ❌ Mockup 上了配色/阴影/圆角 → 拒绝,视觉精细 = 退化成高保真
- ❌ 只画 v1 就进 Phase 2 → 拒绝,行为规则没对齐,高保真阶段必返工

### 反例

- **案例 1**:某后台配置页直接出高保真,PM review 时发现「分组数量」字段该驱动下面表格的列数,但高保真已经写死 3 列 → 返工半天。教训:v1 就能暴露这个联动问题。
- **案例 2**:Mockup v1 敲定了结构就进高保真,高保真 review 时才发现「适用位置的选中会同步到所有分组」这条行为规则没对齐,PM 和前端理解不一致 → 高保真改了一轮半。教训:行为规则要在 Mockup v3-v4 阶段显式标注,不能留到高保真。

## Phase 2:帧覆盖清单(10 类必备帧)

**🎯 目标**:每个列表页/表单页必须覆盖以下 10 类帧。

### 列表页必备帧

| # | 帧类型 | 说明 |
|---|---|---|
| 1 | 默认态 | 有数据正常展示 |
| 2 | 初始空态 | 无数据 + 主 CTA 引导 |
| 3 | 筛选无结果 | 有数据但不匹配 + "清除筛选" |
| 4 | 加载中 | 骨架屏 |
| 5 | 批量操作 | 多选 + 选中行高亮 |
| 6 | 全选 | 所有 checkbox checked |
| 7 | 操作确认 | 1 个完整帧展示 modal 位置 |
| 8 | 操作反馈 | Toast(白底实色,不遮操作区) |
| 9 | 异常告警 | 页面常驻 banner(不用弹窗) |
| 10 | 其余 modal | 独立画出,排在完整帧下方 |

### 关键区分

- **初始空态 ≠ 筛选无结果**:后端逻辑不同,引导操作不同(一个是"创建第一个",一个是"清除筛选")
- **异步事件用 banner 不用弹窗**(弹出时机不可控,用户不一定在线)
- **Toast 白底实色,不要半透明**

**🚧 Phase 2 门禁**:
- ✅ 10 类帧每类都有(不涉及时明确标注)
- ✅ 初始空态和筛选无结果拆分
- ❌ 只画"默认态"就交 → 拒绝,研发无法实现异常路径

## Phase 3:组装红线

**🎯 目标**:按红线组装,杜绝返工元凶。

### 组装红线

- **所有视觉元素必须是组件实例**,0 手搭(layout 容器除外)
- 操作按钮严格按 PRD 状态→操作映射
- 按钮样式层级:primary 填充 = 主 CTA → danger 文字 = 危险 → ghost 文字 = 普通
- **空态必须拆分**:初始空态 ≠ 筛选无结果
- **异步事件用 banner 不用弹窗**
- **Toast 白底实色**

### 按钮尺寸规则

| 场景 | 尺寸 |
|---|---|
| 页面级 CTA | default |
| 筛选区辅助操作 | small |
| 表格行内操作 | small |
| Modal 操作 | default |

### 改 Master 后必做

1. `findAll` 扫描所有该组件的实例
2. 检查 override 是否指向旧值
3. 统一更新
4. **不要假设改了 master 实例就会跟着变**

**🚧 Phase 3 门禁**:
- ✅ 0 手搭视觉元素(所有都是组件实例)
- ✅ 按钮尺寸按场景对
- ✅ 改 master 后已扫 override
- ❌ 遇到 override 冲突就 detach → 拒绝,detach 断了和组件库的关联,后续组件升级追不上

## Phase 4:自检 + 准备 review

**🎯 目标**:原型交给 design-review 前,设计师自己先跑一遍基础检查。

### 自检清单

- [ ] 10 类帧齐全
- [ ] 所有视觉元素都是组件实例
- [ ] 状态机每个转换都有对应帧
- [ ] 文案和 PRD 术语表一致
- [ ] 已砍功能从设计稿彻底移除

### 交付给 review 的物料

- Figma 链接(确保有权限)
- 帧覆盖清单 checklist(哪些已画,哪些标"不涉及")
- PRD 交叉引用(每个帧对应 PRD 哪一章)

**🚧 Phase 4 门禁**:
- ✅ 自检 5 项全过
- ❌ "剩下的 review 时再说" → 拒绝,review 应该找系统性问题,不是 PM 帮你找低级错误

## 常见踩坑

| 坑 | 解决 |
|---|---|
| 第一版原型可用度 ~40% | 正常,需要 10+ 轮 review,耐心 |
| 组件库没 review 就出原型 | 原型会混乱,返工严重;永远:先 review 组件库再设计 |
| 跳过 Mockup 直接出高保真 | 结构改动成本 10 倍,典型场景:字段联动/分组数量驱动布局;永远:先 Mockup 对齐结构再上色 |
| Mockup v1 就当终稿 | 第一版只有结构,没敲定联动/行为/默认值;需迭代到 vN(通常 3-5 轮),每轮聚焦一个维度 |
| Mockup 画得太精细 | 视觉精细(配色/阴影/圆角)禁止,信息精细(示例数据/行为说明)允许,别混淆 |
| 下拉值未填充 | 原型就要填真实的候选值,不是 "<xxx>" |
| 状态标签混乱 | 每个帧名明确标状态:"TaskList/Default"、"TaskList/Filtered-Empty" |
| Token 消耗巨量(300K+/次) | 接受,复杂页面如此;不接受的话做框架先,细节开发时补 |
