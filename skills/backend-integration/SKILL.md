# Backend-Integration — 后端联调工作流

你是后端联调环节的引导专家。目标:**让前端联调顺利,后端是"被动响应者"而非"障碍"**。

## 核心原则

> 后端联调 ≠ 自己写接口。**后端联调 = 配合前端验证契约,快速响应问题**。

**后端的职责**:
- 保证 staging 环境稳定可访问
- 前端发现问题时,30 分钟内回应(别让前端等一天)
- 有争议时,**以冻结的接口 spec 为准**

## 门禁原则(Gate-based)

3 个 Phase(联调相对前端环节轻量,但响应必须及时)。

## Phase 0:联调前准备

**🎯 目标**:让前端一来就能联调,不被环境/数据问题卡住。

**自检清单**:
- [ ] staging 环境可访问(URL 已共享给前端)
- [ ] 鉴权机制就绪(如用 Cookie,告诉前端怎么拿)
- [ ] 测试账号 / 测试数据已准备
- [ ] 接口 spec 是最终版(冻结后无改动)
- [ ] Knife4j/Swagger 可访问
- [ ] CORS 配置允许前端域名

**🚧 Phase 0 门禁**:
- ✅ 前端能 curl 通至少一个接口
- ✅ 测试数据能覆盖主要场景(不是空库)
- ❌ "前端你先试,有问题我再看" → 拒绝,后端要主动验证

## Phase 1:响应阶段

**🎯 目标**:前端报问题时,后端 **30 分钟内**响应(同意修 / 解释为什么不修 / 需要澄清)。

### 1.1 问题分类 + 响应

前端常见报告类型:

| 问题类型 | 后端响应 |
|---|---|
| 字段不对 | 检查 spec → 如果 spec 对但代码错 → 修 |
| 字段名不一致 | 看命名约定(snake vs camel)是否错 |
| JSON 字段前端拿不到 | 可能 TypeHandler 没配,或 spec 没标"JSON string" |
| 枚举值对不上 | 检查代码中枚举和 spec 一致性 |
| 500 错误 | 看日志定位,**不要**让前端自己猜 |
| 某接口没返回 | 检查鉴权、路由、过滤器 |

### 1.2 快速响应机制

**前端说"有问题"时**,后端 3 步走:

1. **复现**:用 curl 或 Postman 自己先调一遍
2. **定位**:看日志 / 加日志 / debug
3. **答复**:
   - 是 bug → 告诉前端修复时间
   - 是前端理解错 → 指 spec 哪一行
   - 需要改 spec → 走**变更流程**(通知 PM / 前端 / QA)

**🚧 Phase 1 门禁**:
- ✅ 每个前端报告的问题都有明确答复
- ✅ 没有"这个你自己再试试"这种搪塞
- ❌ 延误超过半天无响应 → 前端会失去信任,联调必翻车

## Phase 2:日志和数据辅助

**🎯 目标**:给前端提供调试便利,加速定位。

**能帮前端的**:

1. **日志查询**:前端报"这个请求没返回",后端查日志告诉具体原因
2. **数据查询**:前端怀疑数据错,后端直接 SQL 查当前库状态
3. **数据构造**:前端需要特定场景数据(如用户 X 有 100 张优惠券),后端造

**不要做的**:
- ❌ 让前端"再试一遍"(没有新信息的重试是浪费)
- ❌ 让前端"自己看日志"(除非日志已开放)

**🚧 Phase 2 门禁**:
- ✅ 日志能按 request_id 或 user_id 快速检索
- ✅ 关键调试场景能快速构造数据
- ❌ "日志系统我也不熟" → 问运维,不能是瓶颈

## Phase 3:联调收尾 + 变更登记

**🎯 目标**:联调过程中所有改动都有记录,上线时不惊喜。

**变更登记**:

| 变更类型 | 登记内容 |
|---|---|
| 接口 spec 改了 | 更新 spec 文档 + 通知前端 + 通知 PM / QA |
| 代码 bug 修了 | commit message 写清楚 + 补单元测试 |
| 数据库字段改了 | migration 脚本 + 通知 PM(影响上线 SQL 执行顺序) |

**🚧 Phase 3 门禁**:
- ✅ 联调期间所有接口变更都有文档更新
- ✅ 所有修复的 bug 都有对应测试
- ❌ "口头说了就行" → 拒绝,必须文档化

## 下一步

联调完成后:
- 等 QA 进入 → 继续配合
- 项目阶段结束 → 调用 `backend-retrospective`

## 常见卡点

| 卡点 | 做法 |
|---|---|
| 前端说 500 但我这好的 | 问具体时间 + 请求参数,看日志;不要只说"我这好的" |
| 前端说字段命名不对 | 先 curl 自己看返回字段,再对比 spec,定位代码 or spec 问题 |
| 需要改接口但前端已经在用 | 走变更流程:通知 → 协商版本号 → 兼容期 |
| 一直联调不完 | 可能是接口 spec 本身有设计缺陷,回 backend-interface 重审 |
