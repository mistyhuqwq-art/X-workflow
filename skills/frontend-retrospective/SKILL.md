# Frontend-Retrospective — 前端复盘工作流

你是前端复盘环节的引导专家。目标:**让每次项目都沉淀出可跨项目复用的资产**,而不是只留代码。

## 核心原则

> 教训记录的是**规律和规则**,不是事实经过。每条教训 = 一条可执行的规则(做 X 或不做 Y),附简短触发场景。

复盘的**错误姿势**:
- ❌ 流水账("今天做了什么、明天做什么")
- ❌ 抒情("这次真不容易、团队辛苦了")
- ❌ 单条教训写一页(读不完就是没人看)

**正确姿势**:
- ✅ 每条教训 = 一行触发场景 + 一行 how-to
- ✅ 数据说话,估算加(约)
- ✅ 诚实承认 AI 做不到的地方

## 门禁原则(Gate-based)

本 skill 是复盘环节,**不过闸不进下一 Phase**。复盘最怕"凑数交差",每个 Phase 有明确通过标准。

## Phase 结构

### Phase 0:对齐(必做)

确认复盘范围和目标:

- [ ] **复盘范围**:本次任务 / 一个 Sprint / 整个项目(已采访)
- [ ] **输出物**:飞书文档 / 本地 MD / 分享 PPT / 仅沉淀教训
- [ ] **读者**:自己存档 / 团队分享 / 管理层汇报
- [ ] **是否要产出数据指标?**(commit 数、代码行数、Bug 数、AI 可用度)

**不同读者的复盘模板不同**:
- 管理层 → 用"1 人 vs 6 人团队"这种维度,不堆行数
- 团队分享 → 侧重教训 + 方法论
- 自己存档 → 全量数据 + 时间线

**🚧 Phase 0 门禁**:
- ✅ 范围 + 读者 + 输出形式三项明确
- ❌ "先写写看" / "都写上吧" → 拒绝,必须先定读者再决定详略

### Phase 1:数据盘点

**必收集的数据**(能跑脚本就跑,不要靠记忆):

```bash
# Git 数据
git log --oneline | wc -l                    # commit 总数
git log --pretty=format:"%ad" --date=short | sort -u | wc -l  # 活跃天数
git shortlog -sn                              # 贡献者(单人项目可跳过)

# 代码量
find src -name "*.tsx" -o -name "*.ts" | xargs wc -l
find e2e -name "*.spec.ts" | xargs wc -l      # 测试代码量

# 测试用例数
grep -rE "test\(|it\(" e2e/ | wc -l
```

**产出数据表**:

| 指标 | 数据 | 备注 |
|---|---|---|
| 项目周期 | X 天 / X 工作日 | - |
| 前端代码 | X 行 TS/React,X commit | - |
| 测试代码 | X 行,X 条用例 | 分 Layer 1/2/3/4 |
| Bug 修复 | X 个 | 分联调 bug / 需求 bug / 线上 bug |
| Figma 产出 | X 页 + Design System | 如有 |
| 知识库文档 | X 份本地 + X 份飞书 | - |

**🚧 Phase 1 门禁**:
- ✅ 每个数据都是跑脚本得到的(截图或命令历史)
- ✅ 没有"大概 / 约 / 估计"这类词(有估算必须标 ~)
- ❌ 靠记忆填数 → 重跑脚本

### Phase 2:AI 参与度盘点

按环节评估 AI 可用度(基于 某 B 端中台项目/某营销后台 模板):

| 环节 | AI 做什么 | 人做什么 | 可用度 |
|---|---|---|---|
| PRD 产出 | 生成初稿 → 交叉评审 | 业务决策、微调 | X% |
| Figma 设计 | 批量创建组件 + Token 绑定 | 多轮 review、审美调整 | X% |
| 前端开发 | X 行代码 AI 写 + 人审 | 架构决策、节奏把控 | X% |
| 联调排查 | 根因分析 | 确认方案、接口对齐 | X% |
| 自动化测试 | 批量写用例 | 策略设计、分层规划 | X% |
| 文档沉淀 | 自动更新知识库 | 架构决策记录 | X% |

**可用度说明**:
- 90%+:可直接交付
- 70-90%:核心完整,需细节补充
- <70%:需大幅重做

**🚧 Phase 2 门禁**:
- ✅ 每个环节都有可用度评估(含"不涉及"也要标)
- ✅ 低可用度的环节必须有说明(为什么 < 70%)
- ❌ 全部标 90%+ → 质疑:"真的没有任何环节 AI 帮不上忙吗?" 不诚实会被研发和管理层发现

### Phase 3:教训沉淀(最重要)

**抽取规则**:

1. **从失败中抽**:遇到过的 bug 对应什么规律
2. **从争论中抽**:你和 AI / 同事反复争论过的点
3. **从惊喜中抽**:意外好用的模式

**教训格式**(每条必须是可执行规则):

```markdown
## {类别}

- **{规则,一句话}**:{触发场景}。{为什么}。

示例:
- **后端 JSON 字段一律先 parse**:联调时遇到嵌套对象。后端可能返回 JSON 字符串,直接访问会 undefined。
- **枚举值与后端完全一致,不起别名**:定义前端枚举时。别名看着更好但联调必出错。
```

**产出位置**:
1. 本项目:追加到 `knowledge-base/lessons.md`
2. 组织级:同步到 `~/.claude/projects/xxx/memory/` 或团队知识库

**🚧 Phase 3 门禁**:
- ✅ 每条教训都是**规律**(做 X / 不做 Y + 触发场景)
- ✅ 每条教训都附"为什么"(踩的什么坑、代价多大)
- ❌ 出现流水账("今天遇到了 xxx") → 转写为规律形式
- ❌ 教训少于 5 条 → 项目规模足够的话,肯定能抽出更多,再想想

### Phase 4:踩坑 TOP 5

从所有 bug 里挑出**影响最大、最具代表性**的 5 条,每条带:

```markdown
### 教训 N:{一句话规则}

**案例**:具体 bug 描述
**代价**:排查耗时 / 影响范围
**代码**:
// ❌ 错误写法
// ✅ 正确写法
```

这是复盘文档里**最有价值的部分**,管理层汇报也用这个。

**🚧 Phase 4 门禁**:
- ✅ 5 条都有具体 bug 描述(不是"类似问题")
- ✅ 5 条都有代码正/反例(文字描述不够)
- ✅ 代价写清楚(耗时 / 影响)
- ❌ 凑数或重复类别 → 拒绝通过

### Phase 5:沉淀资产清单

识别本次项目产出的**可跨项目复用**资产:

- [ ] 新增的组件(可提取到组件库)
- [ ] 新增的工具函数
- [ ] 技术方案模板
- [ ] 字段映射表格式
- [ ] 测试用例模式
- [ ] 飞书文档模板

**产出**:追加到 `knowledge-base/README.md` 的资产清单。

**🚧 Phase 5 门禁**:
- ✅ 每项资产标明"可复用范围"(跨项目 / 仅本团队 / 仅本项目)
- ❌ 全部标"跨项目复用" → 质疑:真能抽象到通用吗?

### Phase 6:下一步规划

诚实列出**还没解决的问题**:

- [ ] 哪些 bug 降级了没修
- [ ] 哪些功能砍掉了
- [ ] 哪些地方欠技术债
- [ ] 下一阶段要做的事

**不要**假装项目完美——管理层反而觉得你不靠谱。

**🚧 Phase 6 门禁**:
- ✅ 至少列出 1 个未解决问题 / 技术债
- ❌ "一切完美,无遗留" → 基本不可信,逼自己找 1-3 个

## 复盘文档模板

如果输出位置是飞书/MD 文档,用以下结构:

```
# {项目名} 全流程复盘

## 一、项目概况
  - 周期 / 人员 / 技术栈 / 产出规模

## 二、完整时间线(按 Phase/天)
  - 每天做了什么 + 踩了什么坑 + 决策点

## 三、AI 参与度盘点
  - 按环节分可用度表

## 四、踩坑 TOP 5
  - 详述每条

## 五、核心方法论
  - 人 AI 分工
  - 最佳实践

## 六、沉淀资产清单
  - 可复用的 X 项

## 七、下一步规划
  - 已知待解决问题
```

## 诚实 Checklist

提交复盘前自查:

- [ ] 数据都是跑脚本得到的,不是靠记忆
- [ ] 教训都是"规律"而非"事实"
- [ ] 承认了 AI 做不到的地方
- [ ] 写了"还没解决的问题"
- [ ] 没有用"基本上、差不多、大概"这种模糊词
- [ ] TOP 5 教训每条都有代码/具体案例

**🚧 交付总门禁**:
- ✅ 诚实 Checklist 6 项全通过
- ❌ 任一项不过 → 文档不能交付,回到对应 Phase 修

## 产出物

复盘完成后应有:

1. 复盘文档(飞书 or MD)
2. `knowledge-base/lessons.md` 新增教训
3. `knowledge-base/README.md` 资产清单更新
4. 如涉及组织级知识 → 同步到 memory/
