# 发布 / 推仓库类教训

> 针对"把工作流、skill 包、文档推到公开仓库"这类场景的专题教训。

---

## #1 推公开仓库前必做隐私扫描 🔒

**触发场景**:任何文档 / skill 包 / 工作流要 push 到公开 GitHub 仓库前。
**为什么**:内部链接、项目名、公司关键词一旦进公开仓库,**git 历史永久保留**。即使后来删了当前文件,clone 的人 `git log -p` 依然能看到全部历史。

**真实代价**:workflow-dlc 首次 push 就泄漏:
- 18 处飞书文档链接
- 17 个文件的内部项目名(某 B 端中台项目/某营销后台)
- 多处公司关键词(小米/hyperos/mioffice)

**修复代价**:删远程仓库 → 新建 → 重推干净版。所有已经 fork 或 clone 的人都拿到了带历史的版本。

### 强制扫描脚本(push 前跑)

```bash
# push 前必跑这个扫描,有任何输出就不许 push
grep -rEi "mi\.feishu|www\.feishu|{你公司名}|{内部系统名}|{项目内部代号}" . \
  --include="*.md" --include="*.yml" --include="*.json" \
  2>/dev/null | grep -v ".git"
```

## #2 脱敏要用"方法论 + 通用描述"替换

**触发场景**:清理敏感内容时容易"一刀切删掉",丢失重要方法论。
**为什么**:内部项目名("某 B 端中台项目 V1 跑了 927 条测试")本身是**信息含量高的案例**,简单删除会把教训一起丢。正确做法是**保留方法论,脱敏具体指代**。

| 敏感类型 | 脱敏方式 |
|---|---|
| 内部项目名 | "某 B 端中台项目"(保留规模和类型) |
| 内部文档链接 | 改"见维护者私有 registry"(保留引用意图) |
| 具体团队名 | 删除或改"企业内部" |
| 具体人名 | 删除或匿名"{PM}"、"{后端}" |
| 内部系统 URL | 改占位符 |

**关键**:读者仍能理解"为什么这样做",只是不知道"谁做的"。

## #3 分层管理:公开仓库 vs 私有位置

**触发场景**:设计 workflow / skill 包的文档架构时。
**为什么**:所有文档混在一起 = 不清楚哪些能公开、哪些不能。推送前挑敏感内容的工作量会爆炸。

**架构原则**:

```
公开仓库/
  └── 方法论 + 工作流骨架 + 通用模板
      (别人看了能学,不含组织信息)

私有位置/
  └── 真实 source-registry + 真实项目名 + 真实链接 + 实测数据
      (你自己用,不 push 公开)
```

**具体做法**:
- 公开:`sources/source-registry.md` 是模板(字段定义 + 分级方法)
- 私有:`~/Documents/workflow-dlc-private/source-registry-actual.md` 填真实内容
- 两边 schema 一致,但内容分开维护

## #4 新仓库 > filter-repo > 改当前文件

**触发场景**:发现敏感内容已经 push 了,要决定怎么清理。
**为什么**:不同方案隐私保护程度差异巨大。

| 方案 | 隐私彻底度 | 成本 | 已 fork/clone 的人 |
|---|---|---|---|
| **删远程仓库 + 新建同名 + 干净 push** | 🟢 最彻底 | 中(需联系已 clone 的人) | 需要手动切换 |
| **git filter-repo 重写历史 + force push** | 🟡 高 | 高(复杂操作) | 已 fork 的包仍有旧历史 |
| **只改当前文件(新 commit)** | 🔴 **无效** | 低 | git log 照样能看到全部泄漏 |

**绝不允许方案 3**。如果已 push → 选方案 1 或 2,优先方案 1。

## #5 防泄漏 Checklist(push 前)

- [ ] 全文搜索过:公司名 / 内部项目名 / 内部系统 URL / 内部人员名
- [ ] `grep {敏感关键词} .` = 0 匹配(每个都试)
- [ ] 示例代码里的数据是虚构的(非真实用户名 / 邮箱 / token / ID)
- [ ] 配置文件(`.env` / `config.json` / `credentials`)没意外 git add
- [ ] `.gitignore` 覆盖了日志 / 缓存 / 临时文件
- [ ] **git 历史** 也是干净的(新仓库优先;若复用旧仓库,先 filter-repo)
- [ ] 双人复查(自己容易有盲区)

## #6 用 git hook 自动防泄漏(终极方案)

**触发场景**:团队里多人会往公开仓库 push 时。
**为什么**:指望每个人 push 前都手动扫描,迟早出事。用 git hook 强制。

**pre-commit hook 示例**(`.git/hooks/pre-commit`):

```bash
#!/bin/bash
# 敏感关键词黑名单
BLACKLIST="mi\.feishu|www\.feishu|{公司名}|{内部系统}|{项目内部代号}"

# 扫描暂存区
LEAKS=$(git diff --cached --name-only | xargs grep -lEi "$BLACKLIST" 2>/dev/null)

if [ -n "$LEAKS" ]; then
  echo "🚨 发现敏感内容!以下文件包含黑名单关键词:"
  echo "$LEAKS"
  echo ""
  echo "请脱敏后重新 commit,或加 --no-verify 绕过(不推荐)"
  exit 1
fi
```

装法:
```bash
chmod +x .git/hooks/pre-commit
# 或用 husky 工具管理(能被 git clone 的人自动安装)
```

---

## 💡 核心原则一句话

> **公开仓库是永久记录。一旦泄漏,成本是"删仓 + 重建 + 通知所有 clone 的人"**。
>
> 宁可多花 5 分钟扫描,不要后来花 5 小时重建。
