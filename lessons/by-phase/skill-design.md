# Skill 设计教训

> 设计 workflow-dlc skill 时的可执行规则。每条来自真实踩坑。

---

## #1 高置信度 ≠ 可以跳过用户确认

**触发场景**:设计 AI 路由 / 自动判断类 skill 时。
**为什么**:"声明式"= AI 宣告、用户服从,AI 越自信用户越被迫接受。用户永远有权否决 AI 的判断。
**怎么做**:无论置信度多高,都要给用户"确认 or 纠正"的机会,只是形式不同:
- 高置信:一键 Yes(4 选 1:对/全重选/只改角色/只改环节)
- 中置信:标准确认(默认选中最高分)
- 低置信:完全自选

```
# ❌ 反例(原始 M1 workflow-start 设计)
if confidence >= 0.85:
    announce_judgment()
    proceed_to_interview()  # 直接进下一步

# ✅ 正例(修复后)
if confidence >= 0.85:
    announce_judgment()
    ask_confirm_or_reject()  # 一键 Yes,但必须问
```

**来源**:2026-04-23 M1 测试,X 直接问"在判断我的身份之后有给我留自己选择的空间吗?"——暴露 Bug #0。

## #2 门禁标准必须配操作指引

**触发场景**:写"🚧 门禁"式的质量门时。
**为什么**:只写"必须 XX 才能过"但不写"怎么证明 XX",用户会用口头承诺过门。真实场景"配好了,相信我"就把门闯了。
**怎么做**:每个门禁必须附**三类东西**:
- 验证方式(怎么证明过了)
- 通过标准(什么结果算过)
- 拒绝标准(什么不算过)

```markdown
# ❌ 反例
✅ 环境策略已配置可跑通

# ✅ 正例
✅ 环境能跑通,通过以下任一方式证明:
  A. curl 主要接口拿到 200/401(连接通)
  B. 浏览器 Network 截图有 API 响应
  C. 用 Mock 场景,标记 env_mode=mock
不接受:"我配好了,相信我" / "dev server 能起来" / "昨天跑过"
```

**来源**:2026-04-23 M1 测试 Bug #1。

## #3 门禁要分场景,别一刀切

**触发场景**:写 Phase 门禁时,容易只想到"标准场景"。
**为什么**:真实用户可能在 Mock 联调、半 Mock 半真实、纯离线测试 …… 一刀切的门禁要么过严(Mock 时过不了)要么过松(真实联调时放水)。
**怎么做**:
- Phase 0 先把"场景标签"记下来(真实/Mock/混合)
- 后续 Phase 门禁按场景标签分档
- 宽松场景要留"待补"标记,不要假装通过

```markdown
# ❌ 反例
✅ Network 面板人工看过

# ✅ 正例
根据 Phase 0 记录的 env_mode 分档:
- 真实联调: Network 必看
- Mock 联调: Network 标"待补",写入 todo.md
- 混合: 每个接口分别标
```

**来源**:2026-04-23 M1 测试 Bug #2。

## #4 MVP 范围外不拒绝,给降级出路

**触发场景**:M1/M2 阶段性 skill 包,用户可能选到未实现的场景。
**为什么**:直接说"M1 不支持"是堵路,用户没得选。给降级出路让用户感觉"工具在帮我",不是"工具在限制我"。
**怎么做**:
- 在入口 skill 启动时声明范围(让用户提前知道)
- 用户选到范围外 → 列出可用的降级工具(模板 / 通用 skill / 文档)
- 记入日志 `routed_skill: "fallback"`,用于优先级判断

```
# ❌ 反例
"M1 暂不支持 PM·需求阶段,其他角色 M2 发布"(堵路)

# ✅ 正例
"M1 没有自动化 PM skill,但有现成资产:
  - templates/prd-template.md 直接用
  - socratic-dialogue skill 帮你拆解
要我准备好吗?"(给路)
```

**来源**:2026-04-23 M1 测试 Bug #3。

## #5 用虚拟项目跑一遍是验证 skill 的最强手段

**触发场景**:新 skill 写完、还没上真实项目用。
**为什么**:自己读 skill markdown 看不出问题,真实用户走一遍会把隐藏假设全暴露。造虚拟项目成本低(1 小时),收益是"避免在真实项目上翻车"。
**怎么做**:
1. 造一个**结构逼真**的虚拟项目(CLAUDE.md + knowledge-base + src + git log)
2. 模拟用户触发 skill,**严格按 skill 指令一步步执行**(不能跳)
3. 每一步都记录:哪里卡了、哪里模糊、哪里需要用户信息但 skill 没问
4. **让用户扮演真实用户**,可以"不按默认选"
5. 把卡壳点写成 Bug,修完再跑一遍

**虚拟项目要有的元素**:
- ✅ 符合 CLAUDE.md §5 结构(knowledge-base/ + tasks/todo.md + CLAUDE.md)
- ✅ src/ 有一定代码量(至少 5-10 个文件)
- ✅ git log 有 6+ 条真实感的 commit
- ✅ 故意埋 2-3 个"典型 bug"(如 JSON 字段当对象访问)
- ✅ todo.md 有进行中项
- ❌ 不要造太完美,完美项目暴露不了 skill 问题

**收益实证**:2026-04-23 本次测试,1 小时造虚拟项目,发现 4 个 Bug(其中 1 个致命)。

## #6 Skill 的设计文档本身也要留反例

**触发场景**:写 SKILL.md 时只写"正确做法"。
**为什么**:AI 执行 skill 时容易走回常见错误模式(比如未验证声明完成)。写反例能让 AI 意识到这是禁区。
**怎么做**:关键规则都附"❌ 不接受的说法" / "反例"。

示例:
```markdown
# ✅ 正面规则
所有数值必须 4 维完整:值 + 单位 + 区间开闭 + 可配置性

# ❌ 反例(帮 AI 识别禁区)
"长按触发选中"(缺单位、区间、可配置性)
```

## #7 每个自动判断都要写日志,不记不知道哪里错

**触发场景**:AI 路由类 skill,凭信号做决策。
**为什么**:AI 判断错了、用户纠正了,如果没记录,下次还会犯同样错误。三层经验库的"原始层"就是干这个用的。
**怎么做**:每次判断写 JSON 日志,**必含**:
- 信号源(读了什么文件/命令)
- 候选分数(每个候选多少分)
- 置信度
- 用户是否覆盖(`user_overrode: true/false`)
- 最终路由的 skill

```json
{
  "confidence": 1.0,
  "user_confirmed": {"role": "PM", "phase": "需求"},
  "user_overrode": true,    // 关键字段,用来优化后续判断
  "auto_judgment": {"role": "前端", "phase": "联调"}
}
```

有了 `user_overrode: true` 的记录,AI 汇总后就能发现"当 signal 是 fix(api) 密集时,30% 的用户其实是 PM 在验收,不是前端在联调"。
