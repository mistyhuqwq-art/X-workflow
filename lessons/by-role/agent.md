# Agent 设计师教训(12 条)

> 来自 One Click 国际化 CMS Agent 方法论 + workflow-dlc 自身设计实践

---

## 场景选型类

### #1 有现成后台就做内嵌 Copilot
**触发**:启动新 Agent 产品时。
**为什么**:后台页面 = 天然上下文,意图识别难度降一个量级,3 步确认替代 6 步对话。
**规则**:IM 机器人仅用于跨系统查询;配置/操作类场景做内嵌。

### #2 场景不超过 3 个,先做好 1 个
**触发**:想让 Agent 支持多种场景时。
**为什么**:贪多 = 每个都做不好,用户体验差。
**规则**:P0 只做 1 个端到端跑通,P1 扩 1-2 个。

### #3 用户画像必须具体
**触发**:"所有人都能用"这种宽泛目标时。
**为什么**:画像不具体 = 产品定位模糊,没人会真用。
**规则**:画像到"海外运营,老员工,每日使用"这种颗粒度。

## 交互设计类

### #4 用结构化卡片,不用纯对话
**触发**:设计 Agent 返回结果时。
**为什么**:纯文字消息需要用户打字修改,结构化卡片可以点选。
**规则**:Agent 响应 = 带下拉/日期组件的结构化卡片。

### #5 3 步确认链路是铁律
**触发**:设计交互流程时。
**为什么**:超过 3 步 = 用户放弃,直接走后台原有流程。
**规则**:自然语言 → 填表高亮 → 提交,不多一步。

### #6 快捷按钮不用 LLM 动态生成
**触发**:设计上下文快捷操作时。
**为什么**:LLM 生成 = 延迟 + 不确定,用户点击不响应。
**规则**:前端规则映射(页面类型 × 选中状态),硬编码条件分支。

### #7 三色高亮系统
**触发**:Agent 填入表单后。
**为什么**:用户需要知道哪些是 AI 填的、哪些自己改过。
**规则**:浅蓝(AI 填) / 浅黄(AI 填已修改) / 浅红(校验失败)。

## 经验学习类

### #8 日志是 P0 必做,不能延后
**触发**:想"先上线再说,日志后补"时。
**为什么**:补日志 = 重做,P1 经验库依赖 P0 日志。
**规则**:全字段日志(geo/compliance/user_modified/tags)P0 必做。

### #9 user_modified 是经验沉淀的金矿
**触发**:日志字段设计时。
**为什么**:AI 填了什么 vs 用户改成什么 = 所有优化方向的根据。
**规则**:必记录 AI 填写值和用户最终提交值,供模式层分析。

### #10 AI 不能自动沉淀规则
**触发**:想让 Agent 从日志自动学规则时。
**为什么**:AI 会误判个例为规律,累积错误规则污染经验库。
**规则**:AI 只到模式层(提假设),规则层必须人工确认。

### #11 规则层必须能过期
**触发**:经验库运行半年后。
**为什么**:过期规则持续误导,越来越噪。
**规则**:3 个月未命中标记"待复核",双周会决定保留或废弃。

## 分期规划类

### #12 P0 不求完美,求"不留架构债"
**触发**:P0 阶段取舍时。
**为什么**:P0 完美 = 延期;P0 偷工减料 = P1 返工。
**规则**:P0 每个功能做刚好能用的 MVP,但架构基座(协议、日志、规则框架)必须到位。
