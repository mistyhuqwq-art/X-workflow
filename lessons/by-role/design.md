# 设计师教训(16 条)

> 来自 design-workflow-spec v4.0 附录 A 13 条规律 + 实战补充

---

## Token / 组件系统类

### #1 Token 三层架构不可跳级
**触发**:建设计系统时。
**为什么**:组件直接引用 Primitives = 品牌色切换时要改所有组件。
**规则**:Primitives → Tokens → Components,只改 Tokens 就全变。

### #2 组件绑定必须 0 硬编码
**触发**:组件 review 时。
**为什么**:硬编码颜色 = 主题切换/品牌更新时全需手改。
**规则**:0 硬编码色值 / 0 散落字体 / 0 未绑定圆角。

### #3 organism 里所有视觉元素必须来自 atom 实例
**触发**:搭复杂组件时。
**为什么**:organism 里手搭 = atom 升级追不上。
**规则**:organism = 复合 atoms 实例,不自绘。

### #4 primary 和 accent 值相同也不合并
**触发**:初期 primary 和 accent 颜色一样。
**为什么**:语义不同,未来可能分离。
**规则**:保留两个变量,即使值相同。

## 原型组装类

### #5 10 类帧必齐
**触发**:列表页设计时。
**为什么**:漏了任何一类(如筛选无结果),研发无法实现异常路径。
**规则**:默认/初始空/筛选无结果/加载/批量/全选/确认/反馈/告警/modal 全画。

### #6 初始空态 ≠ 筛选无结果
**触发**:画空状态时。
**为什么**:后端逻辑不同,引导操作不同(一个创建第一个,一个清除筛选)。
**规则**:两个帧独立画,引导文案也不同。

### #7 异步事件用 banner 不用弹窗
**触发**:设计告警/通知时。
**为什么**:弹窗出现时机不可控,用户不一定在线。
**规则**:异步提示用页面常驻 banner,弹窗只给同步动作用。

### #8 Toast 白底实色
**触发**:设计反馈 Toast 时。
**为什么**:半透明 Toast 看不清,影响可读性。
**规则**:白底 + 实色,不要半透明。

## 审查 / 一致性类

### #9 改 Master 后必扫 Override
**触发**:改组件属性后。
**为什么**:override 优先级高于 master,改了 master 实例不跟上。
**规则**:findAll 所有实例 → 检查 override → 统一更新。

### #10 三角审查:复用/可实现/可理解
**触发**:深度 review 时。
**为什么**:只看技术合规会漏大问题(组件能复用但没用、研发做不了、用户看不懂)。
**规则**:每个设计方案问自己:组件库有现成吗?研发有坑吗?用户会错过吗?

### #11 不归因于"渲染缓存"
**触发**:截图审查发现异常时。
**为什么**:"缓存问题"是偷懒答复,真问题会掩盖。
**规则**:截图异常必须列出具体节点让用户确认。

## 响应式类

### #12 大屏 ≠ 等比放大
**触发**:做响应式适配时。
**为什么**:大屏可用空间多,等比放大显得空旷。
**规则**:利用空间展开省略文字 / 加宽间距 / 加次要列。

### #13 先做 1 帧验证,再全量
**触发**:响应式铺开时。
**为什么**:全量做完发现方案不对 = 工作量翻倍。
**规则**:选 1 个代表帧做示意,对齐后再全量铺。

### #14 detach 后必重绑 Text Style
**触发**:需要 override 不到的地方(如内部列宽)。
**为什么**:detach 断了和 Style 的关联,字体会乱。
**规则**:detach 立即 findAll TEXT 重绑 textStyleId。

### #15 帧结构必须 auto-layout + FILL
**触发**:响应式容器设计。
**为什么**:FIXED 布局的子节点 resize 后不适应。
**规则**:Sidebar FIXED,Main Content FILL,Table FILL。

## 协作类

### #16 不替用户决策
**触发**:有多个合理选项时。
**为什么**:设计师替用户决定 = 用户后来发现另一个更好,返工。
**规则**:列选项 + 每项影响,等用户回复。

## Figma API 编程类

### #17 FILL 必须在 appendChild 后设置
**触发**:用 Figma Plugin API 给节点设置颜色/填充时。
**为什么**:节点未加入文档时设置 fills 无效,父子关系建立后才能生效。
**规则**:
```javascript
parent.appendChild(node); // 先 append
node.fills = [{ type: 'SOLID', color: { r, g, b } }]; // 再设 fills
```

### #18 Plugin API 不支持 TypeScript 语法
**触发**:在 Figma Plugin 脚本里写 TypeScript。
**为什么**:Plugin API 运行环境是纯 JS,TS 类型注解会报错。
**规则**:Plugin 脚本用纯 JS 编写,或在本地用 tsc 编译后再粘贴到插件环境。

### #19 Variable scopes 必须精细化,禁止 ALL_SCOPES
**触发**:创建 Figma Variable 时。
**为什么**:`ALL_SCOPES` 会把变量应用到所有场景,导致颜色变量错误地出现在圆角等属性中。
**规则**:颜色变量只绑 FILL/STROKE,间距变量只绑 GAP/PADDING 等对应 scope。

### #20 setBoundVariableForPaint 的 base paint 必须用实际色值
**触发**:用 Plugin API 绑定颜色变量到节点时。
**为什么**:base paint 留空或用错误颜色 → 绑定后渲染出错误颜色。
**规则**:创建 paint 时必须传入变量的解析后实际色值作为 base,再 setBoundVariable 绑定语义变量。

## 组件库质量类

### #21 Tab 导航和 FilterChip 语义不同,不能互用
**触发**:搭 organism 级组件时,看到两者外观相似。
**为什么**:Tab 是单选导航(切换视图),FilterChip 是多选过滤(缩小范围),语义完全不同。
**规则**:导航用 tabs_status,内容过滤用 chip_filter,混用会让用户产生操作困惑。

### #22 组件初版 review 后才能用于原型
**触发**:组件库刚建好,想立即开始出原型时。
**为什么**:组件库未 review 直接出原型 → 原型引用错误组件 → 交付后开发返工。
**规则**:组件库经 review(可用度 80%+ )后才进入原型阶段,两步串行。

### #23 COMPONENT_SET 不支持 createInstance
**触发**:用 Plugin API 对 ComponentSet 调用 createInstance() 时。
**为什么**:ComponentSet 是组件集合容器,不能直接实例化,必须先取到具体 Component。
**规则**:先 `componentSet.children` 找到具体的 Component 节点,再调用 `.createInstance()`。

## 设计规范落地类

### #24 颜色必须集中管理,禁止散落 hex
**触发**:向代码文件写入颜色值时。
**为什么**:硬编码 hex 散落各处 → 品牌色更新时逐文件改,必然遗漏。
**规则**:所有颜色集中到 `colors.ts` 或 ConfigProvider,组件只引用语义变量,不直接写 hex。

### #25 可变字体需用原生字重值,不用标准 Web 值
**触发**:为设计稿或代码配置可变字体字重时。
**为什么**:标准 Web 值(400/500/600)在可变字体渲染偏细,视觉效果不符合设计规范。
**规则**:查字体的原生字重轴范围,用实际 CSS font-variation-settings 值(如 330/380/520/630)。
